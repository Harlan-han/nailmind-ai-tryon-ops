# 产品架构规划

## 1. 文档目标
本文件描述甲感 NailMind 的产品架构、能力分层、模块边界与实施建议。重点是产品级和系统级结构，而不是底层技术细节。

## 2. 产品架构总览
甲感 NailMind 采用“双端产品 + AI 中台 + 数据闭环”的结构：
- 用户端负责试戴与决策
- 运营端负责趋势与经营动作
- AI 中台负责生成、推荐、分析与建议
- 行为数据在两端之间形成闭环

## 3. 四层架构

### 3.1 体验层
直接面向用户和运营的产品界面。

#### 用户端体验层
- 首页与发现
- 试戴流程
- 推荐与决策
- 收藏与预约意向
- 个人记录

#### 运营端体验层
- 数据总览
- 趋势分析
- 爆款候选
- 冷门预警
- 款式管理
- AI 日报
- 建议中心

### 3.2 智能能力层
支撑核心 AI 能力。

#### 核心能力
- 手部识别与预处理
- 美甲款式迁移 / 试戴生成
- 款式标签理解
- 推荐逻辑
- 趋势识别
- 运营建议生成

### 3.3 内容与运营层
支撑平台持续运营的业务能力。

#### 核心能力
- 款式素材管理
- 风格标签管理
- 展示状态管理
- 热门与专题配置
- 建议采纳与调整记录

### 3.4 闭环反馈层
负责把用户行为沉淀成运营信号，再反哺用户端体验。

#### 核心闭环
用户试戴与收藏行为 → 趋势识别 → 运营建议 → 款式/内容调整 → 用户继续试戴与转化

#### 当前实现补充
- 用户试戴、收藏、预约意向已沉淀为款式级运营信号
- 运营 Agent 可读取款式曝光、试戴、收藏、预约和转化试戴图
- Agent 当前支持诊断“为什么热”“试戴高但预约低”“收藏高但预约低”“哪些试戴图带来预约”
- 建议仍只进入建议中心等待人工确认，不自动执行运营动作

## 4. 核心模块边界

### 4.1 用户端边界
用户端负责：
- 吸引进入试戴
- 帮助高效选款
- 呈现试戴结果
- 帮助做决策
- 承接收藏与预约意向

用户端不负责：
- 深度商家管理
- 复杂交易履约
- 运营规则配置

### 4.2 运营端边界
运营端负责：
- 看见趋势变化
- 管理款式与标签
- 识别爆款与冷门
- 接收 AI 建议并进行人工决策

运营端首阶段不负责：
- 全自动执行所有线上运营动作
- 完整 BI 平台化能力

### 4.3 AI 中台边界
AI 中台负责：
- 生成试戴结果
- 提供推荐与解释
- 识别趋势变化
- 输出运营建议

AI 中台不应直接替代业务规则。首版应以“AI + 明确规则”组合为主，而不是完全黑盒。

## 5. 核心产品回路

### 回路一：用户决策回路
发现款式 → 发起试戴 → 查看结果 → 获得推荐 → 收藏 / 候选 / 预约

### 回路二：运营判断回路
查看趋势 → 识别机会或问题 → 接收建议 → 调整配置 → 观察后续变化

### 回路三：双端闭环回路
用户行为数据 → 趋势信号 → 运营动作 → 更优分发 → 更高质量用户行为

## 6. 推荐架构思路
推荐体系不应一开始追求复杂建模，而应分阶段：

### 第一阶段
- 标签推荐
- 相似款推荐
- 热门补充推荐
- 简单解释理由

### 第二阶段
- 场景化推荐
- 偏好画像推荐
- 更细的人群适配推荐

### 第三阶段
- 更个性化的多目标排序
- 结合趋势和个体偏好的动态推荐

## 7. 运营智能架构思路

### 第一阶段
- 看板 + 热度识别 + 日报建议

### 第二阶段
- 冷门预警 + 建议中心 + 更丰富的动作建议

### 第三阶段
- 更强的运营 Agent
- 节日与专题建议
- 商家视角建议

### 当前实现状态
- 已接入 DeepSeek 作为语言模型层
- 已将运营看板、趋势、漏斗、建议中心、款式转化诊断封装为只读工具
- Agent 输出包括回答、证据、行动建议、工具调用轨迹，并支持同步到建议中心

### 当前演示版边界与后续增强
- 当前演示版的登录以本地验证码为主，适合体验验收；正式上线需替换为短信服务、邮箱登录、风控和账号资料管理。
- 当前端外 Agent 以模拟通道和 Webhook 形态证明可接入性；正式版需补齐飞书/企微/QQ 的真实机器人配置、验签、重试和消息归档。
- 当前运营 Agent 不直接改运营配置，只生成建议和待确认动作卡；正式版应增加可审计执行、权限分级、回滚与操作日志。
- 当前用户端试戴助手先服务选款推荐和咨询信号沉淀；正式版应接入更完整的用户偏好记忆、候选清单对比、多轮问答和运营侧聚合洞察。

## 8. 推荐实施技术方向（高层建议）
为了方便 AI 协同开发，建议采用前后端分离与 AI 服务分层的结构：
- 用户端 Web：适合快速试戴体验验证
- 运营后台 Web：适合趋势分析与管理
- 后端服务：负责业务逻辑与任务调度
- AI 服务层：负责图像处理、推荐与运营建议

推荐技术方向可以采用：
- 前端：Next.js
- 后端：FastAPI
- 数据库：PostgreSQL
- 缓存 / 任务状态：Redis
- 图片与结果存储：对象存储

这部分是实施建议，不是当前项目文档的唯一约束。

## 9. 架构拆分原则
1. 把用户端、运营端、AI 能力、内容管理拆清楚
2. 先做清晰模块，不先做过度平台化抽象
3. 中台能力要可替换，避免被某个模型能力绑死
4. 任何增强能力都不能破坏核心闭环的简洁性

## 10. 实施优先顺序
1. 跑通用户端试戴决策主链路
2. 跑通运营端趋势感知主链路
3. 跑通推荐与建议链路
4. 再扩展候选对比、冷门预警、款式管理
5. 最后增强个性化与 Agent 化能力

## 11. 架构完成标准
一个阶段的架构只有在以下条件下才算成立：
- 该阶段的用户端体验完整可用
- 该阶段的运营端价值完整成立
- AI 能力不是孤立能力展示，而真正服务模块
- 双端之间存在明确的反馈关系
