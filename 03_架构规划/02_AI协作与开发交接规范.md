# AI 协作与开发交接规范

## 1. 目标
本文件用于指导多个 AI 工具协作推进甲感 NailMind，确保后续开发不会因为上下文割裂而反复重做。

## 2. 当前协作前提
用户计划主要使用 Claude Code、Kimi 2.6、Hermes Agent、Codex 等工具推进本项目。因此文档需要承担统一事实来源的作用。

## 3. 各类 AI 更适合承担的工作

### Claude Code
适合：
- 主工程搭建
- 跨文件功能实现
- 前后端对接
- 文档与实现一致性维护

### Kimi 2.6
适合：
- 长文档整理
- 原型说明文案
- 功能细化补充
- 路演或业务表达材料

### Hermes Agent / Codex
适合：
- 局部功能并行实现
- 样式与页面层拆分
- 特定模块快速补齐

## 4. 协作原则
1. 所有 AI 先读文档再做任务
2. 单轮只推进一个阶段或一个模块，不并发重规划
3. 如果某 AI 发现文档冲突，先修文档，不直接各自实现
4. 一切实现要能回溯到 PRD、架构或版本路线图中的某一条

## 5. 推荐协作流程

### 阶段 A：产品与原型阶段
- 使用本目录中的文档继续补线框图、页面说明、交互细节
- 不急着下沉到接口与表结构

### 阶段 B：骨架搭建阶段
- 由 Claude Code 搭项目骨架与主目录
- 明确用户端与后台端的结构
- 搭建试戴链路最小闭环

### 阶段 C：核心闭环实现阶段
- 实现用户端上传、选款、试戴、推荐、预约意向
- 实现后台总览、趋势、爆款、日报
- 确保双端闭环能跑通

### 阶段 D：增强与优化阶段
- 增加候选对比、冷门预警、款式管理、建议中心
- 补偏好画像、场景化推荐、更多运营建议

## 6. 交接时最少需要给下一个 AI 的信息
1. 当前推进到哪个 Phase
2. 本轮完成了哪些模块
3. 哪些文档已更新
4. 哪些内容明确还没做
5. 下一步最推荐推进什么

## 7. 文档更新规则
当发生以下变化时，必须同步更新文档：
- 产品范围改变
- 页面结构改变
- 版本阶段改变
- 核心闭环改变
- 技术实现方向发生重大变化

## 8. 当前不建议过早细化的内容
在没有进入正式开发之前，不需要优先展开：
- 复杂数据库设计
- 复杂埋点体系
- 过细 API 文档
- 复杂权限系统

原因不是这些不重要，而是现在更关键的是保证产品结构、阶段路线和交接清晰。

## 9. 推荐交付顺序
1. 先基于现有文档完成原型或视觉框架
2. 再搭用户端试戴主链路
3. 再搭运营后台主链路
4. 再补推荐、候选、对比、冷门等增强模块
5. 最后再深入到指标、埋点、细粒度优化

## 10. 当后续开始写代码时
应补充新增文档或更新 `CLAUDE.md`，至少包括：
- 实际目录结构
- 启动命令
- 测试命令
- 关键服务说明
- 开发阶段中新的约束
